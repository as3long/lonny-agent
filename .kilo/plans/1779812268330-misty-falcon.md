# Plan: Replace @opentui/core with @earendil-works/pi-tui

## Overview

Replace `@opentui/core` (v0.2.15) with `@earendil-works/pi-tui` (v0.75.5) in the TUI module. The new library is a pure TypeScript differential-rendering TUI library with a simpler component model (no flexbox, no yoga-layout). Layout must be handled manually.

## API Mapping

| @opentui/core | @earendil-works/pi-tui |
|---|---|
| `createCliRenderer(options)` | `new TUI(new ProcessTerminal())` |
| `renderer.root` (RootRenderable) | `TUI` itself (extends `Container`) |
| `BoxRenderable` | `Box` (padding + background only; no flexbox) |
| `TextRenderable` | `Text` |
| `InputRenderable` | `Input` (has `onSubmit` callback; Emacs-style keybindings built-in) |
| `ScrollBoxRenderable` | N/A — scrolling handled by TUI's differential rendering (content clipped to terminal height) |
| `SelectRenderable` | `SelectList` (different API: `onSelect`, `onSelectionChange`, `getSelectedItem()`) |
| `MarkdownRenderable(content, streaming, syntaxStyle)` | `Markdown(text, paddingX, paddingY, theme, defaultTextStyle?)` — no streaming mode |
| `renderer.on('resize', cb)` | Handled by `TUI` internally; use `terminal.columns` / `terminal.rows` |
| `renderer.destroy()` | `tui.stop()` |
| `SyntaxStyle` | `MarkdownTheme` (function-based styling with ANSI) |
| Flexbox layout (`flexGrow`, `flexDirection`, `width: '30%'`) | Manual layout — must implement custom `SplitLayout` for side-by-side panels |

## Key Differences

1. **No flexbox** — pi-tui only stacks children vertically in a `Container`. For the horizontal split (chat + side panel), we need a custom `SplitLayout` component.
2. **No streaming markdown** — `Markdown` component doesn't support streaming. We'll accumulate content and call `markdown.setText()` on each `write()` call.
3. **No `SelectRenderable.getSelectedOption()`** — use `SelectList.getSelectedItem()`.
4. **`Input` has built-in Emacs keybindings** and an `onSubmit` callback — no need to manually wire `onKeyDown`.
5. **`TUI` handles input routing** to the focused component automatically.
6. **No `renderer.setTerminalTitle()`** — use `terminal.setTitle()`.
7. **Side panel visibility** — since there's no `visible` property, the `SplitLayout` will conditionally render the right panel.

**Note on resize handling for side panel visibility:**
Since pi-tui doesn't expose a resize event directly, we can use `tui.addInputListener()` and check `terminal.columns` on each input, or use a polling interval. A simpler approach: add an input listener that toggles side panel visibility when terminal width crosses the threshold.

**Note on chat markdown streaming:**
Since `Markdown` doesn't support streaming mode, we'll call `markdown.setText(content)` on each `SessionOutput.write()`. This will cause a re-render. To avoid excessive re-renders, we can batch writes (debounce).

## Risks

1. **No streaming markdown** — without streaming, each token update triggers a full re-render of the markdown component. For long LLM outputs, this may cause flicker. Mitigation: debounce `setText` calls (e.g., 50ms interval).
2. **No ScrollBox** — the chat area isn't inside a scrollable container. The TUI's differential rendering only shows the bottom `terminal.rows` lines. If the chat content exceeds terminal height, older messages scroll off. This is acceptable behavior for a chat TUI.
3. **`SelectList` requires theme** — unlike `SelectRenderable` which had sensible defaults, `SelectList` needs a `SelectListTheme` object. We'll create a minimal one.
4. **`Input` has Emacs keybindings** — Ctrl+A/E/B/F/P/N/D/K/Y etc. are built-in. This is fine for a chat input but differs from the previous behavior.
5. **Windows VT input** — `ProcessTerminal` already handles `enableWindowsVTInput` for modified keys on Windows, which is good.
